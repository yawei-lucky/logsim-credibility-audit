# HUGSIM CF-O 002 可控遮挡纠错复验预注册

## 为什么需要 002

CF-O 001 的六次仿真按预注册完成，几何状态检查 37/37 帧通过，但 RGB 支撑
37/37 帧均被报告为 `unavailable`。排查确认车辆实际已出现在 `CAM_FRONT`；问题
是分析器把 HUGSIM `obj_boxes` 的第三坐标直接当作相机正向 z，而该字段保存的是
渲染器高度的相反数，导致目标投影框落在真实车辆下方。

002 只修正这一个坐标约定：全局 box 转换到 vehicle frame 时使用
`z_vehicle = z_ego - z_actor`。横向/纵向转换、六条件配置、36 step、RGB 阈值
10、最小参考像素 25、2% 非嵌套入口、背景稳定门槛和全部裁决规则均与 001 保持
不变。新增单元测试直接覆盖该符号约定。

## 执行边界

六个条件使用与 001 相同的配置，但全部写入新的 `run002`；不得复用或覆盖 001
产物。分析器继续核对本预注册提交、配置哈希、运行提交、干净工作树、step 和
plan-writer 契约。

002 是方法纠错复验，不是独立场景或独立统计样本。由于坐标问题是在查看 001
缺测后发现的，即使 002 通过，整体证据仍为 `down-weighted`。允许的最强主张和
禁止主张完全沿用 001：最多支持当前场景、相机、资产、轨迹和两档遮挡中的内部
几何与目标 RGB 支撑方向，不支持真实传感器、AD 响应、real–sim 等效或 HUGSIM
整体可信性。
