# HUGSIM CF-I interaction capability gate 001 — corrected review

## Question

Does the available HUGSIM code contain a controller in which one vehicle's
state or planned motion changes another vehicle's output? Two independently
scripted trajectories do not satisfy this definition.

## Method

The probe used HUGSIM revision `adeca402cad4af8635e13d0a105e2fee6a14de85`.

- `IDM`: the same synthetic neighbor was moved from off the responder's route
  to 6 m ahead on the route while responder state and route stayed fixed.
- `AttackPlanner`: the responder used the first released extreme-scenario
  state `[x=0, y=-15, yaw=0, v=3]`. The ego was placed 15 m or 5 m ahead with
  identical heading and speed. Both future inputs were generated by HUGSIM's
  own `constant_headaway`, as in the full planner loop. `best_k=1` removed
  random top-k selection, and an identical-input repeat checked determinism.
- The released time grid (`ego prediction index dt=0.25 s`, AttackPlanner
  candidate index `dt=2/19≈0.1053 s`) was compared with an aligned
  `0.1053/0.1053 s` control. Although `plan.py` passes `dt=0.1 s`, the spline
  generator spans `20×0.1=2.0 s` with 20 points including both endpoints.
- Controller interfaces and output states were recorded by
  `scripts/audit_hugsim_interaction_capability.py`.

This is a controller-level deterministic probe. It is not yet a rendered scene
experiment and does not use HUGSIM RGB, semantics, depth, or scores as truth.

The corrected review replaces the earlier hand-built, independently varying
ego-future input with scene-loop-reachable inputs generated from current ego
states.

## Interpretation

The internal capability question and the released scene-loop qualification now
have different answers:

- **Internal mechanism found:** HUGSIM contains controllers whose output changes
  under a reachable change in another vehicle's state. The response remains in
  the aligned-time-grid control.
- **Released scene-level gate not passed:** the local released configuration
  predicts ego states every `0.25 s`, but its AttackPlanner candidate indices
  are about `0.1053 s` apart, then compares them at matching array indices.
  Those indices do not describe matching future times, so interaction timing
  is not qualified.
- Earlier multi-car runs made only from independent `ConstantPlanner` actors
  remain rejected as interaction evidence.

The two usable mechanisms have different boundaries:

- `IDM` is neighbor-responsive, but returns the unchanged state when no route
  is available. A scene-level IDM experiment therefore requires a qualified
  route/map path.
- `AttackPlanner` consumes the ego's predicted future and can generate a
  response without the HD map in the current implementation. Its objective is
  adversarial proximity, not ordinary human traffic behavior.
- In the complete HUGSIM planner loop, that predicted future is a
  constant-heading extrapolation of the current ego state, not the external
  AD's full future plan. The ordinary-neighbor input is also passed as
  `neighbors[1:]`; generic non-ego vehicle-to-vehicle response is therefore not
  qualified here.
- `plan.py` stores a single planner-wide `ATTACK_FREQ`. A one-responder test
  avoids claiming that distinct per-actor response timings are honored.

Overall evidence is `down-weighted`. The probe establishes code-path
sensitivity and exposes a scene-loop timing defect; it does not establish
causal timing inside a complete HUGSIM rollout, realistic merging/yielding, physical
feasibility over time, sensor consistency, AD response, or real-world fitness.

## Indicator result

This audit's main research product is the first CF-I indicator candidate, not a
HUGSIM repair:

| Indicator | Type | Measurement | Known negative | Positive control | Narrow qualification |
|---|---|---|---:|---:|---|
| `CF-I-T1` stimulus-response indexed-time alignment | hard constraint | maximum timestamp offset between states compared at the same trajectory index | released `0.25/0.1053 s` grids: `2.75 s` maximum offset, `rejected` | aligned `0.1053/0.1053 s` grids: `0 s`, `accepted` | control discrimination `accepted` |

The indicator is qualified only to detect this class of time-grid mismatch. It
does not measure whether the response is realistic, safe, or representative of
human driving. The released mismatch remains negative evidence; the aligned
case is a validation control, not a replacement result used to make HUGSIM look
successful.

## Next bounded experiment

Use the released mismatch as a negative control and an aligned configuration as
a positive control. Run one state-only paired rollout with a timed ego stimulus
and test whether three candidate indicators discriminate the controls:

1. `CF-I-T1`: indexed-time alignment;
2. pre-stimulus responder divergence;
3. post-stimulus response latency and state continuity.

Only after those indicators behave correctly should the same states be
rendered. Describe any positive HUGSIM result only as adversarial ego-response
capability; do not generalize it to realistic traffic interaction.
