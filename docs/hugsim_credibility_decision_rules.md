# HUGSIM Credibility Decision Rules

> Purpose: define when HUGSIM closed-loop evidence should be accepted, down-weighted, or rejected during Phase 1 audit.

These rules do not judge HUGSIM globally. They apply to individual closed-loop segments, scenario runs, or metric events.

## Decision Labels

Use exactly these labels:

```text
accepted
down-weighted
rejected
```

## Accepted Evidence

Use `accepted` only when the closed-loop segment has enough simulator-side evidence to support the driving outcome.

Minimum conditions:

- The scene and scenario assets are public or otherwise clearly identified.
- The scenario YAML is recorded.
- Ego state before and after the step is logged.
- Actor states before and after the step are logged.
- The rendered observation is available or indexed.
- The planned trajectory and resulting action are logged.
- Collision / route / metric events are supported by internal geometry.
- No obvious rendering artifact affects the relevant decision region.
- No obvious actor scale, orientation, or occlusion error affects the event.
- The metric event is consistent with temporal evidence across nearby frames.

Typical examples:

- Normal step transition with stable observations and plausible state update.
- Collision event where object geometry, rendered view, and temporal approach are mutually consistent.
- Drivable-area violation where ground geometry, rendered road boundary, and ego trajectory agree.

## Down-weighted Evidence

Use `down-weighted` when the segment is still informative but has credibility weaknesses.

Common reasons:

- Extrapolated view quality is visibly degraded but the core event remains interpretable.
- Actor scale or orientation is slightly unstable but not enough to invalidate the event.
- Semantic/depth evidence is missing, but RGB and internal geometry are consistent.
- The event occurs near reconstruction boundaries.
- Public asset provenance is known, but some preprocessing or conversion details are unclear.
- The metric is computed from internal geometry, but rendered observation evidence is weaker.
- Actor behavior is plausible but generated by a stress-case controller whose assumptions should be noted.

Typical examples:

- A near-miss event where geometry supports the event but occlusion evidence is incomplete.
- A route-completion or comfort score where state logging is complete but rendered evidence is not central.
- A drivable-area warning where exported ground points are sparse but the trajectory trend is clear.

## Rejected Evidence

Use `rejected` when the segment should not support a simulator or AD-agent conclusion.

Common reasons:

- Scenario assets are missing, private, or not identifiable.
- The run cannot be reproduced from recorded paths/configs.
- The relevant actor is visibly malformed, missing, floating, duplicated, or inconsistent across views.
- The collision / near-miss is caused by obvious rendering or geometry artifact.
- The ego or actor state jumps discontinuously without a plausible controller explanation.
- The event occurs outside the reconstructed support region and no confidence warning is available.
- Internal geometry and rendered observation contradict each other in the event region.
- Metric computation fails, silently skips frames, or uses missing/invalid point clouds.

Typical examples:

- Collision is reported by internal geometry but the rendered actor is absent or badly misplaced.
- Drivable-area violation is reported because of a sparse or corrupted ground point cloud.
- A high-risk scenario uses an unavailable private scenario file or untracked asset.

## Per-Event Audit Checklist

For each event, record:

| Field | Required? | Notes |
|---|---|---|
| scenario_yaml | Yes | Path and commit/hash if available. |
| scene_asset | Yes | Scene name and local path / source URL. |
| vehicle_assets | Yes | Actor asset IDs from `plan_list`. |
| ego_before | Yes | Ego box, pose, velocity, steering. |
| ego_after | Yes | Ego box, pose, velocity, steering. |
| actors_before | Yes | Object boxes and controller IDs. |
| actors_after | Yes | Object boxes and controller IDs. |
| observation_rgb | Yes | At least frame index / file reference. |
| observation_semantic | Preferred | Useful for relation consistency. |
| observation_depth | Preferred | Useful for geometry consistency. |
| planned_traj | Yes | AD or dummy trajectory consumed by simulator. |
| action | Yes | Acceleration and steer-rate. |
| metrics | Yes | NC, DAC, TTC, comfort, route completion, hdscore when available. |
| audit_decision | Yes | accepted / down-weighted / rejected. |
| audit_reason | Yes | One or two concrete reasons. |

## Relation-Level Checks

Prioritize task-relevant relations over visual beauty:

- front / rear / left / right stability;
- same-lane / adjacent-lane / off-road consistency;
- approaching / receding consistency;
- occluding / occluded-by consistency;
- actor scale / orientation consistency;
- lane and drivable-area consistency;
- whether risk-increasing events are supported by sensor, geometry, map, and temporal evidence.

## Metric Interpretation Rule

HUGSIM's HDScore and component metrics should be treated as AD-performance metrics under the simulator.

They can support an audit decision only when paired with simulator-credibility evidence. Metric values alone are insufficient to accept a closed-loop result as credible.
